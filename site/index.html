Getting started

Claude Fleet Portal is a locally-hosted mission control for your Claude Code agents. It spawns headless claude -p processes, parses their live event streams and surfaces everything in a single cockpit UI.

Prerequisites

  • Node.js 20+ and pnpm (if building from source)
  • The Claude Code CLI installed globally (optional — a free deterministic mock is included)
  • macOS, Windows or Linux

Quick start

The fastest path is the prebuilt desktop app — download it from the latest release, open it, and you are in. No Node, no pnpm, no checkout needed.

If you prefer building from source:

  • git clone the repo and run ./install.sh
  • Then ./start.sh — the web UI opens at :4318, the control plane listens on :4319
  • Use ./start.sh --mock for a free tour with no API tokens
Tip

The mock mode replays deterministic streams from tools/mock-claude.mjs — great for developing or demoing without spending tokens.

Launching agents

Click the + Launch Agent button in the top header to open the launch dialog. Claude models run through the native Claude Code path; Codex and OpenCode selections route through their enabled engine add-ons automatically.

Configuration options

  • Task prompt — describe what the agent should do, or start with any /command
  • Working directory — the repo or folder the agent operates in
  • Model & effort — select Claude, Codex or OpenCode models from one grouped picker; Claude runs keep the effort dial (low / medium / high / max)
  • Budget ceiling — hard USD limit; the server auto-kills runs that hit it
  • Template — pick from 12 pre-tuned agent profiles (Debugger, Test Writer, etc.)
  • Skills & slash-commands — attach skills from your installed plugins

What happens after launch

The control plane spawns the right CLI process, assigns it a run ID, and begins parsing the event stream. Within seconds the run card appears on your dashboard with live status, engine, model, tokens and available cost tracking.

Orchestration & campaigns

Single agents are powerful — but orchestrated fleets ship faster. The portal supports two orchestration modes:

Subagent trees

When a root agent spawns subagents, the portal automatically reconstructs the full hierarchy. You see the tree in real time: each node shows its own timeline, token burn and status. You can stop, steer or approve any node independently.

Campaigns

Campaigns take orchestration further. Give the portal a single high-level objective and it plans a dependency DAG — one real worker agent per subtask, run in parallel where possible. A synthesizer agent merges the results when all workers complete.

  • Navigate to /orchestrate to create a campaign
  • The planner breaks your objective into subtasks with dependencies
  • Workers execute in parallel, respecting the DAG order
  • The synthesizer collects outputs and produces the final result
Example

Objective: "Refactor the auth module to use JWT tokens". The planner might create subtasks for (1) audit current auth, (2) design JWT schema, (3) implement token generation, (4) update middleware, (5) write tests — each assigned to a worker agent.

Engines & compression

Claude is still the native full-feature engine, but Codex and OpenCode now participate in the same model catalog. If an add-on is enabled, its curated models appear beside Claude models in every picker.

Unified model pickers

  • Launch modal — choose Claude, Codex or OpenCode from grouped model options, with a custom engine model field when needed
  • Templates, schedules and benchmarks — engine-tagged catalog models route to the matching CLI at launch time
  • Kanban cards — assign a model per card; engine models delegate PM build/fix/resolve attempts to Codex or OpenCode

Engine behavior

Engine add-ons are one-shot headless runs with a flat timeline. Stop works; Claude-only features such as resume/input, subagent trees, retry escalation and per-run cost streams stay on the native Claude path.

Headroom compression routing

The Compression add-on can proxy compatible engine traffic too. Codex routes through OPENAI_BASE_URL=http://127.0.0.1:<port>/v1. OpenCode routes through ANTHROPIC_BASE_URL for anthropic/* models and OPENAI_BASE_URL for openai/* models; other providers talk directly to their endpoints.

Live monitoring

Every run streams data to your browser via Server-Sent Events (SSE). No polling, no refresh — the cockpit updates in real time.

Fleet dashboard

The root view shows every run as a card: status, model, effort dial, cost gauge, token burn, subagent count and depth. Filter by status (live / done / failed / killed) or search by task description.

Run detail view

Click any run card to drill into its detail page. You get:

  • Workflow tree — the full root-to-subagent hierarchy with per-node streaming
  • Timeline — chronological view of every tool call, message and decision
  • Span waterfall — duration bars for each operation, like a profiler
  • OTLP telemetry overlay — OpenTelemetry spans when enabled
  • File viewer — browse files the agent has read or written
  • Diff view — see code changes as they happen

Steering controls

You are not a passive observer. At any point you can stop a run, send a follow-up message, approve a pending action, or kill a runaway agent.

Guardrails & budgets

Production fleets need hard limits. The portal enforces them at two levels.

Budget ceilings

  • Per-run ceiling — set when launching; the CLI flag enforces it and the server double-checks with an auto-kill
  • Per-project ceiling — aggregate spend limit for all runs in a project
  • Daily tracking — the sidebar gauge shows cumulative daily spend, heating from green through amber to red

Concurrency & safety

  • Configurable concurrency caps — limit how many agents run simultaneously
  • Validation gates — require checks to pass before results are accepted
  • Merge gates — hold merges for your explicit approval, or run in full-auto mode
Safety first

All data stays on your machine. The portal is local-first by design — your code, API keys and spend data never leave your network.

Autonomous PM

The Project Manager mode turns the portal into an autonomous development pipeline.

How it works

  • Attach a git repository to a project
  • Add cards to the Kanban board describing tasks
  • The PM agent picks up cards, builds in isolated git worktrees, validates the output, and gates the merge
  • Each card can use the PM default or a specific Claude, Codex or OpenCode model
  • PR mode opens pull requests instead of merging directly
  • A conflict-resolution agent handles merge conflicts automatically

Kanban board

The board at /projects/[id]/board shows columns for backlog, in progress, review and done. Cards move automatically as agents work through them. You can assign engine models to specific cards, drag cards to reprioritize, add new ones, or block a card from being picked up.

Templates & skills

Agent templates are pre-tuned profiles that shape how the spawned agent behaves.

Built-in templates

The portal ships with 12 templates: Debugger, Test Writer, Security Auditor, Frontend Builder, Refactor Expert, Documentation Writer, and more. Each template defines a system prompt, default model, effort level and attached skills. Custom templates can now store engine-tagged model IDs too.

Custom templates

Create your own templates at /templates. Define the persona, attach specific skills and slash-commands, set default configuration, and save. Your custom templates appear in the launch dialog alongside the built-in ones.

Skills & plugins

Every skill and slash-command from your installed Claude Code plugins is surfaced in the portal. Attach them to individual runs or bake them into templates. Launch a run directly on any /command — the portal handles the rest.

Self-updates

The portal keeps itself current automatically.

How updates work

  • The server checks GitHub Releases in the background on a regular interval
  • When a new version is found, an amber dot appears on the Releases nav entry
  • Click Update to vX.Y.Z — the server runs git pull --ff-only and pnpm install
  • Dev watchers reload automatically; in production, a simple restart picks up the changes
  • Desktop builds point you at the new installer download instead

Version tracking

The Releases page at /releases shows the installed version, current commit SHA, latest release, update status, and the full changelog for every published release.

or open it once → System Settings → Privacy & Security → “Open Anyway” → launch again
